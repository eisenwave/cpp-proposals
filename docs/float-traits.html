1. Introduction

The information provided by std::numeric_limits is insufficient for various applications.

Say we want to decompose a std::float32_t into its 22+1 mantissa bits, into its biased exponent, and into its sign bit. Doing so is helpful when converting e.g. from std::float32_t to a custom big_int or big_decimal arbitrary-precision type because it effectively requires constructing a big_int from the mantissa, then shifting it to the left or right depending on the exponent.

  • std::numeric_limits doesn't inform about the location of the sign bit, but at least we can obtain it from std::signbit.
  • std::numeric_limits doesn't inform about the location or amount of the exponent bits or about the exponent bias, but at least we can obtain the exponent using std::frexp or std::ilogb.
  • std::numeric_limits doesn't inform about the location of mantissa bits and it doesn't specify whether there is a preceding implicit bit or not. std::frexp is also of limited use because it returns a floating-point number with magnitude in range [0.5,1). For a conversion to big_int, we could still use std::trunc to truncate the original value and cast to an integer; the conversion to big_float is more complex.

While it is possible to compensate for the lack of information by calling a variety of <cmath> functions, this is cumbersome and may be much slower if the compiler doesn't treat every such function as an intrinsic but actually emits a call frexp instruction. If std::numeric_limits provided more information, we could simply bit-cast the floating-point value and obtain all the necessary information.

FLT_HAS_SUBNORM allows to query whether float has subnormal numbers since C++17, but std::numeric_limits doesn't provide this information.

std::numeric_limits is not future-proof for decimal floating-point types such as _Decimal128 introduced in C23. Notably, an implementation of ISO/IEC 60559 decimal floating-point numbers can either use

  • decimal encoding (aka. DPD — Densely Packed Decimal), in which case the mantissa is a sequence of 10-bit declets, or
  • binary encoding (aka BID — Binary Integer Decimal), in which case the mantissa is simply an integer.

See also ISO/IEC 60559:2020 §3.5.

A portable workaround for this problem is to bit-cast a decimal floating-point number and inspect the bit pattern to identify how it is encoded.

std::numeric_limits provides limit information about floating-point attributes specified in ISO/IEC 60559, such as preferredWidth attributes, value-changing optimization attributes, and reproducibility attributes. It only provides information about the rounding-direction attributes in the form of round_style.

See also See also ISO/IEC 60559:2020 §4.

For instance, [P3375R3] Reproducible floating-point results considers adding a fundamental or library type with reproducible floating-point semantics similar to Java's strictfp types, but std::numeric_limits could not be used to query whether a type is reproducible or not.

A key issue with std::numeric_limits is that it cannot be extended in the standard because it can be specialized by the user, so adding any new members would retroactively make those user-defined specializations invalid.

2. Design

To solve all these problems, I propose a new std::float_traits class template that provides more information about floating-point types than std::numeric_limits. This class template also cannot be specialized by the user, so that we don't end up with another non-extendable traits type that necessitates std::floating_traits, std::floating_point_traits, std::floating_point_number_traits, etc. in the future.

The synopsis looks as follows, where information not provided by std::numeric_limits is marked:

template<class T> struct float_traits { // Range static constexpr T normal_min; static constexpr T normal_max; static constexpr T normal_lowest; // Precision and radix static constexpr int digits; static constexpr int digits10; static constexpr int max_digits10; static constexpr bool is_signed; static constexpr int radix; static constexpr T epsilon; static constexpr T round_error; // Exponents static constexpr int min_exponent; static constexpr int min_exponent10; static constexpr int max_exponent; static constexpr int max_exponent10; // Special values static constexpr bool has_negative_zero; static constexpr bool has_infinity; static constexpr bool has_negative_infinity; static constexpr bool has_quiet_NaN; static constexpr bool has_signaling_NaN; static constexpr bool is_signaling_NaN_quiet; static constexpr bool has_subnormals; static constexpr T infinity; // optional static constexpr T negative_infinity; // optional static constexpr T quiet_NaN; // optional static constexpr T signaling_NaN; // optional static constexpr T subnormal_min; // optional static constexpr T subnormal_max; // optional static constexpr T subnormal_lowest; // optional // Representation static constexpr bool is_iec559; static constexpr bool is_iec559_binary_interchange; static constexpr bool is_iec559_decimal_interchange; static constexpr bool is_densely_packed_decimal; static constexpr int width; static constexpr int significant_bits; static constexpr bool has_explicit_significant; static constexpr int exponent_bits; static constexpr int exponent_bias; static constexpr bool has_sign_bit; static constexpr bool has_unique_finite_representations; static constexpr bool has_non_canonical_representations; // Attributes static constexpr bool traps; static constexpr bool tinyness_before; static constexpr float_round_style round_style = round_toward_zero; };

2.1. List of new members

2.1.1. bool has_negative_zero

If true, -T{0} is negative zero and is a distinct value from T{0}.

2.1.2. bool has_negative_infinity

If true, negative_infinity is defined.

2.1.3. bool is_signaling_NaN_quiet

This variable is true is signaling NaNs are treated as quiet NaNs by the implementation. For example, this is the behavior of GCC when -fsignaling-nans is not specified, and the behavior on some architectures such as WASM.

Notably, it is possible for has_signaling_NaN to be true while is_signaling_NaN_quiet is also true, which means that signaling NaN exists on paper and may be classified as such by issignaling in <math.h>, but operations on signaling NaN do not trap regardless.

2.1.4. bool has_subnormals

true if the type has subnormal numbers.

This information can be queried for float using FLT_HAS_SUBNORM, but that macro is only available since C++17, and std::numeric_limits does not provide this information.

2.1.5. T negative_infinity

Equals −∞.

There is a subtle difference between negative_infinity and -infinity: if the floating-point type only has an unsigned infinity representation, then negative_infinity is not defined, whereas -infinity applies the unary minus operator to infinity, whatever the result of that might be (we don't have wording for that).

2.1.6. T subnormal_max

The greatest subnormal number, and only defined if the type has subnormal numbers.

This is simply a counterpart to subnormal_min, and could also be obtained using std::nexttoward, but doing so is cumbersome.

2.1.7. T subnormal_lowest

Equal to -subnormal_max, except that using the unary minus operator on subnormal_max may result in flushing to zero.

2.1.8. bool is_iec559_binary_interchange

true if the representation of T is an ISO/IEC 60559 binary interchange format, such as binary16, binary32, binary64, or binary128.

This is not simply equivalent to is_iec559 && radix == 2 because is_iec559 is also true for extended precision types such as long double, which are specified by ISO/IEC 60559 but are not interchange formats.

is_iec559_binary_interchange && width == 32 reliably classifies a type as having binary32 representation, which is always true for std::float32_t and typically for float as well.

2.1.9. bool is_iec559_decimal_interchange

true if the representation of T is an ISO/IEC 60559 decimal interchange format, such as decimal32, decimal64, or decimal128.

2.1.10. bool is_densely_packed_decimal

true if T is a decimal floating-point type that uses DPD (Densely Packed Decimal) encoding, rather than using BID (Binary Integer Decimal) encoding.

2.1.11. int width

The total amount of bits in the floating-point representation.

2.1.12. int significant_bits

The amount of bits in the significant of the floating-point representation.

This is essential for decomposing a floating-point number.

2.1.13. bool has_explicit_significant

true if the significant of the floating-point representation is not preceded by an implicit bit.

ISO/IEC 60559 binary interchange formats have an implicit bit, meaning that normal finite numbers have an implicit '1.' preceding the significant, whereas subnormal numbers have an implicit '0.' in front.

However, there are also formats such as x87 80-bit long double, where the leading significant bit is part of the representation.

This information is essential for decomposing a floating-point number because it informs the user whether they need to prepend a 1 or 0 in front of the significant bits, or whether all bits are already present in the representation.

2.1.14. int exponent_bits

The number of exponent bits in the floating-point representation.

This is also crucial for decomposing floating-point numbers.

2.1.15. int exponent_bias

The number added onto the raw exponent when interpreted (as encoded in the representation).

For decimal floating-point types, there are two possible interpretations of exponent bias. Some investigation needed.

2.1.16. bool has_sign_bit

true if there is a dedicated sign bit in the representation of T.

It is possible for is_signed to be true while has_sign_bit is false, because the sign could also be encoded as part of the significant or in some other way, rather than always occupying a dedicated bit.

2.1.17. bool has_unique_finite_representations

true if every finite number has a unique representation in the floating-point type.

It is possible for std::has_unique_object_representations_v<T> to be false because there are multiple alternative representations of the value NaN. This obscures whether finite numbers have a unique representation.

ISO/IEC 60559 binary interchange formats have unique representations for finite numbers, meaning that every possible bit pattern corresponds to a unique value, except for NaN and infinity (although infinities also have unique representations). x87 80-bit long double and ISO/IEC 60559 decimal interchange floating-point types do not have unique representations for finite numbers.

With this information, if a dataset consists entirely of finite numbers, std::memcmp can be used to compare two values for equality.

Negative zero and positive zero are distinct finite values.

2.1.18. bool has_non_canonical_representations

true if the type has non-canonical representations, i.e. representations which are valid but not preferred by the implementation and which typically wouldn't be produced by any arithmetic operation.

ISO/IEC 60559 decimal interchange formats have non-canonical representations, and ISO/IEC 60559 binary interchange formats have none.

If this variable is false, the <math.h> function iscanonical returns true for every argument.

2.2. Choice of header

std::float_traits should be located in <limits>. This makes it easy to upgrade from std::numeric_limits to std::float_traits, and it means that std::float_traits is easily accessible without a heavy include like <numeric>.

3. Implementation experience

None. That is, unless we consider the precedent of std::numeric_limits, which this new feature is based on.
